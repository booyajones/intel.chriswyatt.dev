Three Simple Steps
1. You Share Your Idea
Drop it in however works for you -- a Slack message, a Zoom recording, an email, a ClickUp ticket, even a voice memo. No forms to fill out. No templates required. Just tell us what you're thinking.

AI Assistants What each Claude Project bot does

1
PRD Intake Assistant
For anyone with a product request
Personality: Friendly, efficient, no-nonsense. Respects people's time. Under 10 minutes.

Capabilities

  • Accepts any input: transcripts, emails, tickets, voice memos
  • Two paths: guided walkthrough (6 questions) or fast-track paste
  • Web search to verify stakeholder claims in real time
  • Cross-references Knowledge Base automatically
  • Conversational mode for ambiguous inputs

The 6 Questions

  • Q1: What's the problem? (not the solution)
  • Q2: Who owns this today?
  • Q3: What's the scope?
  • Q4: Ideas on how to fix it?
  • Q5: Biggest assumption?
  • Q6: Anything else Product should know?

2
PRD Builder
For Product Managers
Personality: Precise, thorough, structured. Thinks like a senior PM who has been burned by vague requirements.

Capabilities

  • Takes validated Brief or raw PM input
  • Self-research: web search, Knowledge Base, Confluence
  • Verifies regulatory context (Nacha, PCI, SOC2)
  • Generates 8-section PRD with SCQ problem statement
  • Classifies decisions: reversible vs. irreversible
  • Flags uncertain sections with VERIFY tags

PRD Sections

  • 1. Problem Statement (SCQ)
  • 2. Business Rationale
  • 3. Functional Requirements (MECE)
  • 4. System Reality Check
  • 5. Constraints + Compliance
  • 6. Success Metrics
  • 7. Decisions + Open Questions
  • 8. Partner Context

3
Story Writer
For Engineering Leads
Personality: Engineering-minded, zero ambiguity. Thinks like a senior engineer burned by "the system should work correctly" as acceptance criteria.

Capabilities

  • Takes approved PRD (checks Pipeline Handoff block)
  • Maps P0/P1 requirements to individual stories
  • Given/When/Then ACs (happy + edge + error paths)
  • Sizes stories: S/M/L (one dev, one sprint)
  • Dependency map and sprint ordering
  • PRD compliance check before generating

Story Structure

  • Metadata: Story ID, parent PRD, sizing
  • User Story: As a [persona], I need...
  • Why: Business outcome + cost of not building
  • Current State: System context, code refs
  • ACs: Given/When/Then every path
  • Edge Cases: Bulk, failures, permissions
  • Story Map: Dependencies + sprints

Workflow 6 steps, 2 quality gates

1
Capture
Raw input normalized. Transcripts, emails, tickets, voice memos. Speakers parsed, decisions separated from discussion.
2
Research
AI self-research before asking humans. Web search, Knowledge Base, Confluence. Verify claims. Time-boxed: 5-15 min.
3
Problem Brief
Half-page framing doc (6 fields). Sent to domain SME for confirmation before PRD work begins.
⬥ Gate 1: SME Validation
4
Full PRD Draft
8-section PRD: SCQ, business rationale, MECE requirements (P0/P1/P2), system check, constraints, metrics, decisions.
5
SME Review
5 structured questions. Any feedback format. Conflicts trigger return to Brief. 48-hour timeout.
⬥ Gate 2: PRD Approved
6
Ship
Branding, QA checklist (AI audit), Word doc delivery. User stories downstream with ACs, sizing, dependencies.

Decision Framework Reversible vs. irreversible

Type 1: Reversible
Decide now. Move on.
UI copy, config defaults, notification thresholds, feature flags. PM decides immediately.
Type 2: Irreversible
Deliberate. Then commit.
Data model changes, API contracts, compliance implications. Requires stakeholder sign-off.

Methodologies Minto, Cagan, Bezos, JTBD

Minto Pyramid
SCQ problem statements. MECE requirements. Top-down communication.
Cagan
Problem-first. Separate problem from solution. Empower engineering.
Bezos
Decision classification. Reversible = fast. Irreversible = deliberate.
JTBD
Stories as jobs-to-be-done. Outcome over feature.

Quality Rules 19 rules, zero exceptions

Derived from real failures. AI QA audits every document. View full rules →
01
Transform, Don't Polish
Intake forms are input, not output.
02
PRD ≠ Stories
What/why and how-to-build are separate.
03
Find the Real Persona
Who has the pain, not who filed it.
04
Given/When/Then Only
Bullets are not acceptance criteria.
05
System Context Required
No context = request, not requirement.
06
One Feature Per Story
Different systems = different stories.
07
Explicit Exclusions
State what is NOT in scope.
08
Questions Must Block
Owner, deadline, impact for each.
09
Output Replaces Input
Never paste original "for reference."
10
Quantified Metrics
Baseline, target, method, timeframe.
11
SCQ Every Problem
Situation, Complication, Question.
12
Tag Every Decision
Reversible = now. Irreversible = discuss.
13
Simplest Solution First
Don't propose AI when "lock field" works.
14
Verify Team Assignments
Wrong team = doesn't understand org.
15
Facts vs. Assumptions
VERIFY tags. Cite sources. No guessing.
16
Read Request Literally
Don't expand scope without asking.
17
Flag Secondary Sources
Non-owners may describe symptoms.
18
48-Hour SME Timeout
No response = escalate or proceed.
19
Research Completeness
Fill all 6 fields or mark UNKNOWN.
